🗄️ Solutions 5: SQL & Database (50 Topics)

SQL Queries

1. Difference between WHERE and HAVING 🔥Top
WHERE filters individual rows before grouping. HAVING filters summary groups after the GROUP BY clause.
2. INNER vs LEFT vs FULL Join 🔥Top
INNER (matching both), LEFT (all from left + matching right), FULL (all from both, NULLs for mismatches).
3. Subquery vs CTE 🔥Top
Subquery: Nested query. CTE (Common Table Expression - WITH): Named, more readable, and supports recursion. CTEs are generally preferred for complex logic.
4. Window functions 🔥Top
Perform calculations across a set of table rows that are somehow related to the current row. `ROW_NUMBER()` (unique), `RANK()` (skips ties), `DENSE_RANK()` (doesn't skip ties).
5. Find the 2nd highest salary
`SELECT MAX(salary) FROM emp WHERE salary < (SELECT MAX(salary) FROM emp)` or `SELECT salary FROM emp ORDER BY salary DESC LIMIT 1 OFFSET 1`.
6. Find duplicate rows
`SELECT col, COUNT(*) FROM table GROUP BY col HAVING COUNT(*) > 1`.
7. UNION vs UNION ALL
UNION ALL: Simply joins results (fastest). UNION: joins results and removes duplicates (requires extra sort/dedup).
8. GROUP BY — find department-wise max
`SELECT dept_no, MAX(salary) FROM employees GROUP BY dept_no`.
9. Self-join — managers example
Joining a table to itself. `SELECT E.name, M.name as Manager FROM emp E JOIN emp M ON E.mgr_id = M.id`.
10. COALESCE vs NULLIF
COALESCE returns first non-null value. NULLIF returns NULL if two arguments are equal (prevents divide-by-zero).

Indexing & Performance

11. What is an index? B-Tree vs Hash 🔥Top
A data structure that speeds up retrieval. B-Tree: Standard, supports range searches (<, >). Hash: Extremely fast but only for exact equality (=).
12. Clustered vs Non-Clustered 🔥Top
Clustered: Physically reorders rows in the table; only one per table (usually Primary Key). Non-Clustered: Separate structure from the table that points to the values.
13. When does index hurt?
During **INSERT, UPDATE, DELETE** operations as the index must be updated alongside the data. Also adds storage overhead.
14. What is a covering index?
An index where all the columns in the SELECT clause are present in the index itself. The DB doesn't have to touch the actual table at all (Index-Only Scan).
15. EXPLAIN/ANALYZE 🔥Top
Tool to view the execution plan. Look for **Index Scans** (good) vs **Seq Scans** (bad for large tables). It shows total cost and actual timing.
16. What causes full table scan?
Searching without an index, using `NOT`, `LIKE '%value'` (leading wildcard), or if the index is too fragmented.
17. How to optimize slow query? 🔥Top
1. Use `EXPLAIN`. 2. Add appropriate Indexes. 3. Avoid `SELECT *`. 4. Avoid N+1 queries. 5. Consider Materialized Views for complex aggregations.
18. What is query caching?
The DB stores the results of a specific SELECT query. If the identical query is run again (and data hasn't changed), results are returned instantly from memory.

Database Design & Normalization

19. Normalization (1NF, 2NF, 3NF) 🔥Top
1NF: No duplicate rows/cols. 2NF: No partial dependency on PK. 3NF: No transitive dependency. Aim: Minimize redundancy.
20. When to denormalize? 🔥Top
In Read-Heavy systems (Reporting/Analytics) where performance is more important than space. Reduces the number of joins.
21. Primary vs Unique vs Foreign
Primary: Identity, unique, non-null. Unique: Unique, can have NULL. Foreign: Links to PK of another table to maintain integrity.
22. What are constraints?
Rules enforced on data. `CHECK` (logic, e.g., age > 18), `NOT NULL` (mandatory), `DEFAULT` (fallback value).
23. 1:1, 1:N, M:N relationships
1:1: Unique FK in one table. 1:N: FK in the 'Many' table. M:N: Junction table required (e.g., student_id, course_id).
24. Design a schema for E-commerce
Tables: Users, Products, Categories, Orders (FK to User), Order_Items (Junction between Order and Product), Payments.
25. What is an ER diagram?
Entity-Relationship diagram. Visual representation of tables, their columns, and how they relate (connecting lines).

Transactions & ACID

26. What is ACID? 🔥Top
Atomicity (everything or nothing), Consistency (DB remains valid), Isolation (safe concurrent runs), Durability (persists after crash).
27. Isolation levels 🔥Top
Read Uncommitted (dirty reads), Read Committed (No dirty reads), Repeatable Read (No phantom updates), Serializable (Perfect isolation, slowest).
28. What are dirty, phantom, non-repeatable reads?
Dirty: Reading uncommitted data. Non-Repeatable: Data changes while you read twice. Phantom: New rows appear while you read a range.
29. Optimistic vs Pessimistic locking
Optimistic: Assume no conflict; check version number on update. Pessimistic: Lock the row immediately (SELECT FOR UPDATE).
30. What is a deadlock in DB?
T1 locks R1 and waits for R2. T2 locks R2 and waits for R1. DB aborts one to clear the jam.
31. What are savepoints?
Markers within a transaction that allow you to roll back a specific PART of a transaction without losing everything.

NoSQL & Advanced

32. SQL vs NoSQL 🔥Top
SQL: Fixed schema, ACID, vertical scale. NoSQL: Dynamic, horizontal scale, BASE. Use SQL for Finance, NoSQL for Large Social/Catalog data.
33. MongoDB — document model
Stores data in JSON-like BSON documents. No predefined schema. Collections of documents instead of tables of rows.
34. Redis — use cases 🔥Top
Caching, Session management, Leaderboards (Sorted sets), Pub/Sub, Distributed locking. Extremely fast as it's in-memory.
35. What is eventual consistency?
Guarantees that if no new updates are made, all reads will eventually return the last updated value. Used in distributed systems like Cassandra.
36. PostgreSQL vs MySQL
Postgres: Fully ACID, complex queries, feature rich. MySQL: Simple read/write performance, easier setup, standard for many web apps.
37. Materialized view
Physical copy of view results. Speeds up complex reports significantly at the cost of disk space and stale data (needs manual refresh).
38. Stored procedures vs functions
Procedures: Can return multiple/no values, can call transactions. Functions: Must return one value, only for calculations.
39. Triggers — when and why?
Automatically runs code when INSERT/UPDATE occurs. Used for audit logs or enforcing complex business integrity rules.
40. DB migration — zero downtime
Steps: 1. Add column (allow NULLs). 2. Double-write both columns. 3. Migrate old data. 4. Point app to new column. 5. Drop old column.

Practical SQL Problems

41. Employees earning more than manager
`SELECT E.name FROM emp E JOIN emp M ON E.mgr_id = M.id WHERE E.salary > M.salary`.
42. Running total
`SUM(val) OVER (ORDER BY date) as running_total`.
43. Find consecutive login days
Use `ROW_NUMBER()` and subtract from `date`. Grouping by the resulting "Date Group" identifies sequences.
44. Pivot table
Using `CASE WHEN` to turn row values into column headers: `SUM(CASE WHEN month='Jan' THEN val ELSE 0 END) as Jan`.
45. Delete duplicate rows keeping one
`DELETE FROM table WHERE id NOT IN (SELECT MIN(id) FROM table GROUP BY duplicate_col)`.
46. Find Nth highest salary
`SELECT DISTINCT salary FROM emp E1 WHERE n-1 = (SELECT COUNT(DISTINCT salary) FROM emp E2 WHERE E2.salary > E1.salary)`.
47. Customers who bought all products
`SELECT customer_id FROM orders GROUP BY customer_id HAVING COUNT(DISTINCT product_id) = (SELECT COUNT(*) FROM products)`.
48. Moving average last 7 days
`AVG(price) OVER (ORDER BY date ROWS BETWEEN 6 PRECEDING AND CURRENT ROW)`.
49. Recursive CTE — org hierarchy
`WITH RECURSIVE org AS (SELECT id, mgr_id FROM emp WHERE mgr_id IS NULL UNION ALL SELECT E.id, E.mgr_id FROM emp E JOIN org O ON E.mgr_id = O.id)`.
50. Social media schema 🔥Top
Tables: Users, Posts, Comments (FK Post, User), Likes (Junction User-Post), Follows (Self-joining Junction User-User).
