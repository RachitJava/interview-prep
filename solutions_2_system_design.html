Classic System Design (Must-Know)

1. Design a URL Shortener (TinyURL) 🔥Top
Components: API Gateway, App Service, Hash Generator (Base62), NoSQL DB (for fast lookup), Cache (for hot URLs).
Bottleneck: Collision in Hashing. Solution: Pre-generate unique IDs using a centralized ID Generator (Snowflake).

2. Design Twitter/X 🔥Top
Components: Fan-out Service, Redis Cluster (In-memory Newsfeed), Load Balancer, User/Follow Graph DB (Neo4j or sharded SQL).
Bottleneck: Celebrity Tweet Fan-out. Solution: Hydrate celebrity tweets at read-time (Pull model) vs regular users at write-time (Push model).

3. Design Instagram
Components: S3 (Storage), CDN, Postgres (Metadata), Redis (Feed), Background Workers (Resizing).
Bottleneck: Read heavy feed. Solution: Pre-compute feeds and store in Redis. Use CDN for global edge delivery.
4. Design WhatsApp/Chat System 🔥Top
Components: WebSocket Servers (Bidirectional), Session Manager, Message Store (Cassandra/NoSQL for high throughput).
Bottleneck: Managing 1 million+ persistent connections. Solution: Horizontal scaling of WebSocket servers with a persistent session mapping in Redis.

5. Design YouTube
Components: Transcoder (Encoding), CDN (Content delivery), Object Storage (S3), NoSQL (Comments/Metadata), Recommendation Engine.
Bottleneck: Massive Upload Bandwidth. Solution: Parallel Chunk Upload + Offline Background Processing/Encoding.

6. Design Netflix 🔥Top
Components: Microservices for Catalog, CDN (Open Connect), Adaptive Bitrate Streaming (HLS), Personalization engine.
Bottleneck: Global Latency. Solution: Massive use of local CDNs and local caches inside ISP networks.
7. Design Uber/Ola 🔥Top
Components: Geospatial Index (Quadtree/Google S2), Dispatch Service, Ride Service, Real-time Location Tracking (WebSockets).
Bottleneck: Real-time location updates of 1M drivers. Solution: Quadtree sharding by geo-location; Pub/sub for driver updates to nearby riders.

8. Design Amazon/E-Commerce
Components: Inventory Service, Cart Service, Order Management, Payment Gateway (Third-party), Search (Elasticsearch).
Bottleneck: Flash Sale Concurrency. Solution: Use Message Queues for order processing; Redis-based distributed locking for inventory.
9. Design Google Search
Components: Crawler (Spider), Indexer (Inverted Index), Ranking Engine (PageRank), Result Merger.
Bottleneck: Document Processing Latency. Solution: MapReduce/Spark for distributed indexing of trillions of pages.
10. Design Google Maps
Components: Map Rendering (Tiles), Routing Service (Dijkstra/A*), Traffic Updates (Crowdsourced data).
Bottleneck: Shortest Path Calculation. Solution: Pre-compute paths for major junctions; Hierarchical routing graphs.
11. Design Dropbox/Google Drive 🔥Top
Components: Chunking Service (Block-level), Metadata DB, Sync Service (Notification), Object Storage.
Bottleneck: Large file sync. Solution: Block-level deduplication (only upload changed chunks) + Differential sync.
12. Design Rate Limiter 🔥Top
Components: Token Bucket, Leaky Bucket, Fixed Window, Sliding Window Log, Sliding Window Counter.
Bottleneck: Multi-node Distributed Lock. Solution: Redis LUA scripts to update counters atomically without locking.
13. Design Notification System
Components: Message Queue, Notification Workers, Third Party Gateways (Twilio/Firebase), Retry Logic.
Bottleneck: Failure of 3rd Party APIs. Solution: Dead-letter queues (DLQ) and exponential backoff retry.
14. Design Pastebin
Components: Key generator, Object Storage (Text), Analytics.
Bottleneck: Storage Growth. Solution: TTL (automatic deletion) + Cold storage for old pastes.
15. Design Zoom/Video Conferencing
Components: Signaling Server, SFU (Selective Forwarding Unit), MCU (Multipoint Control Unit), WebRTC.
Bottleneck: Network Jitter/Latency. Solution: Adaptive bitrate switching; SFU instead of MCU to reduce server-side transcoding.

Data-Intensive Systems

16. Design a Cache (Redis) 🔥Top
Components: Eviction Policy (LRU/LFU), TTL, In-memory Storage, Persistence (RDB/AOF).
Bottleneck: Cache Stampede / Thundering Herd. Solution: Mutex/Locking; Prefetching data before it expires.
17. Design a Message Queue (Kafka) 🔥Top
Components: Broker, Topic, Partition, Producer, Consumer.
Bottleneck: Rebalance Overhead. Solution: Static partition assignment or reducing consumer count.
18. Design a Key-Value Store
Components: Consistent Hashing, Replication, Sloppy Quorum, Vector Clocks (Conflict resolution).
Bottleneck: Network Partitioning. Solution: Use Gossip protocol for failure detection.
19. Design a Search Engine
Components: Inverted Index, TF-IDF Ranking, Distributed Indexing.
Bottleneck: High Indexing Latency. Solution: Incremental Indexing via ElasticSearch/Solr logic.
20. Design a Web Crawler
Components: URL Frontier (Queue), IP Fetcher, HTML Parser, Duplicate Remover (Bloom Filter).
Bottleneck: Politeness Policy. Solution: Per-domain queuing; delay timers.
21. Design a Logging System
Components: Log Shipper (Logstash), Indexer (Elastic), Visualizer (Kibana), Buffer (Kafka).
Bottleneck: Massive Log Volume. Solution: Sampling; Compression; Tiered storage.
22. Design a Metrics/Monitoring System
Components: Collector, Time Series DB (Prometheus/InfluxDB), Alerter, Dashboard (Grafana).
Bottleneck: High Cardinality Data. Solution: Downsampling (reducing resolution of old data).
23. Design Typeahead/Autocomplete 🔥Top
Components: Trie Data Structure, Frequency Map, Cache for Top 10 results.
Bottleneck: Searching Trie in real-time. Solution: Pre-compute top results at each Trie node; Use sampling to update frequencies.
24. Design a Recommendation System
Components: Data Collector, Feature Store, Model Trainer, Ranking Gateway.
Bottleneck: Training/Prediction Latency. Solution: Offline training; Online inference using pre-computed embeddings.
25. Design a CDN
Components: Edge Servers, Origin Server, Pull/Push Cache, Global Load Balancer.
Bottleneck: Cache Expiry Consistency. Solution: Immediate invalidation API; Versioned assets (file.v1.css).

Architecture Concepts (Theory)

26. Monolith vs Microservices 🔥Top
Concept: Monolith is easy to deploy/test. Microservices allow independent scale/deployment but add network complexity. Use Monolith for early startups; Microservices for large teams (Sabre/Verizon).
27. SQL vs NoSQL 🔥Top
Concept: SQL (ACID, Fixed Schema) for Transactions. NoSQL (BASE, Flexible) for massive reads or unstructured data (AmEx telemetry).
28. CAP Theorem
Concept: Consistency, Availability, Partition Tolerance. Pick two. In a distributed network (P), you must choose between C (Strict truth) or A (Response always).
29. Consistent Hashing
Concept: Mapping keys to a ring. When nodes change, only K/N keys are moved. Prevents massive cache thrashes. Used in DynamoDB/Redis Cluster.
30. Database Sharding 🔥Top
Concept: Horizontal split of data by key (e.g. UserID). Cons: No more joins across shards; complicated re-sharding.
31. Database Replication
Concept: Master-Slave (Master for writes, Slaves for reads). Improves read scale and redundancy.
32. Load Balancing
Concept: Round Robin, Least Connections. L4 (IP/Port) vs L7 (Application/Header). Nginx is a great L7 example.
33. API Gateway Pattern
Concept: Central entry point for Auth, Logging, Rate Limiting. Aggregates microservices into one interface.
34. Circuit Breaker Pattern
Concept: Prevents cascading failures. If service A is slow, B "opens circuit" and returns error immediately instead of waiting/timing out.
35. CQRS
Concept: Command Query Responsibility Segregation. Writing data and Reading data use different models/DBs for perfect optimization.
36. Event Sourcing
Concept: Instead of storing state (balance=10), store events (+5, +3, +2). Perfect for audit trails in Finance (AmEx).
37. Saga Pattern 🔥Top
Concept: Distributed Transactions. Series of local transactions. If one fails, run "Compensating Transactions" to undo previous steps.
38. Idempotency
Concept: Doing an action multiple times gives the same result. Crucial for payments (sending a unique `request_id`).
39. Long Polling vs WebSockets vs SSE
Concept: WebSockets (Real-time, Bi-directional). SSE (Server-to-Client only). Long Polling (Client waits for server response).
40. OAuth 2.0 / JWT 🔥Top
Concept: OAuth for session delegation; JWT for stateless authentication. JWTs are signed objects that server verifies without DB.
